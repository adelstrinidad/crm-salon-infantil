# Generate Implementation Plans

## How to Use This Prompt

**Purpose**: This prompt orchestrates the generation of technical implementation plans for all enablers and use cases defined in the project specifications.

**When to Use**:
- You have enablers defined in `specs/enablers/`
- You have use cases defined in `specs/use-cases/`
- You want to create implementation plans following the plan template

**How to Launch**:

```
Read the prompt file at .ai/example_prompts/generate-implementation-plans.md and execute it.
Generate implementation plans for all enablers and use cases.
```

**Alternative (specific items only)**:

```
Read the prompt file at .ai/example_prompts/generate-implementation-plans.md and execute it.
Generate implementation plans only for EN-06 and UC-01.
```

---

## Objective

Generate technical implementation plans for enablers and use cases following the plan template structure at `.ai/2_templates/plan-template.md`.

## Input Documents

Read these documents to understand scope:

1. **Enablers Overview**: `specs/enablers/enablers-overview.md` - Lists all enablers with dependencies
2. **Use Cases Overview**: `specs/use-cases/use-cases-overview.md` - Lists all use cases with dependencies
3. **Plan Template**: `.ai/2_templates/plan-template.md` - Structure for implementation plans
4. **Constitution**: `../.ai_project_memory/constitution.md` - Core development principles
5. **Backend Constitution**: `../.ai_project_memory/constitution-backend.md` - Backend-specific principles
6. **Frontend Constitution**: `../.ai_project_memory/constitution-frontend.md` - Frontend-specific principles

## Output Requirements

For each enabler/use case, generate:

```
specs/[enablers|use-cases]/[ID]/
├── plan.md              # Implementation plan (this workflow)
├── research.md          # Phase 0 research findings
└── tasks.md             # Generated by /tasks command (NOT this workflow)
```

**Example**: `specs/enablers/EN-06/plan.md`, `specs/use-cases/UC-01/plan.md`

## Execution Strategy

**Constraint**: Process items sequentially following dependency order.

**Resilience Design**: Stateless and idempotent - checks filesystem before each item.

### Phase 1: Initial Setup

1. **Read overview documents**:
   - Read `specs/enablers/enablers-overview.md`
   - Read `specs/use-cases/use-cases-overview.md`
   - Parse dependency graph to determine processing order

2. **Scan existing plans**:
   - Check `specs/enablers/*/plan.md` for existing enabler plans
   - Check `specs/use-cases/*/plan.md` for existing use case plans
   - Identify which items still need plans

3. **Determine processing order** (dependency-aware):
   ```
   EN-01 → EN-02 → EN-03 → EN-04, EN-05 → EN-06 → EN-07, EN-08
                                      ↓
                                    UC-01 → UC-02 to UC-20
   ```

4. **Optional - Knowledge Graph MCP** (if available):
   - Create `ImplementationPlanning` entity with:
     - Total enablers: count
     - Total use cases: count
     - Dependency order list
     - Status: in_progress

### Phase 2: Iterative Plan Generation

For each item in dependency order:

1. **Check if plan exists**:
   - Skip if `specs/[type]/[ID]/plan.md` already exists

2. **Read item specification**:
   - Read full specification file (e.g., `specs/enablers/EN-06-backend-framework.md`)
   - Extract: components, dependencies, acceptance criteria, technical requirements

3. **Read dependent plans** (if any):
   - Load plans of dependencies to understand interfaces and contracts
   - Extract relevant technical decisions

4. **Execute Plan Template Phases**:

   **Phase 0: Research**
   - Identify unknowns from Technical Context
   - Research dependencies and integration patterns
   - Document findings in `research.md`

   **Phase 1: Design**
   - Extract entities and data models
   - Define API contracts (if applicable)
   - Verify constitution compliance
   - Update `plan.md` with design decisions

5. **Write plan files**:
   - Create directory: `specs/[type]/[ID]/`
   - Write `plan.md` (implementation plan)
   - Write `research.md` (Phase 0 findings)

6. **Optional - Knowledge Graph MCP** (if available):
   - Update item entity with status "planned"
   - Add observation with completion timestamp

7. **Proceed to next item**

### Phase 3: Verification

1. Scan all item directories for `plan.md` files
2. Compare against overview documents
3. Report any missing plans
4. Optional - Generate completion summary via Knowledge Graph

## Learnings & Best Practices

### Acceptance Criteria Format

**Use regular bullets in plan.md, checkboxes in tasks.md**:

| Document | Format | Purpose |
|----------|--------|---------|
| `plan.md` | `- [FR-xxx] Criterion text` | Criteria **definition** |
| `tasks.md` | `- [ ] Task description` | Implementation **tracking** |

**Correct format in plan.md**:
```markdown
### Functional Acceptance
- [FR-1002.1] User can open building dropdown
- [FR-1002.3] Building selection filters sensors
- Building dropdown renders in < 200ms
```

**Incorrect format** (don't use checkboxes in plan.md):
```markdown
- [ ] [FR-1002.1] User can open building dropdown  ❌
```

### NFR vs FR Classification

**Functional Requirements (FR)**: Describe system behaviors and workflows
- Authentication redirect to Azure AD → **FR-1010.3** (workflow behavior)
- User can select up to 10 sensors → **FR-1002.9** (functional capability)

**Non-Functional Requirements (NFR)**: Describe quality attributes
- Zoom operations < 200ms → **NFR-01** (performance)
- Responsive UI during loading → **NFR-05** (responsiveness)
- Extensible codebase → **NFR-07** (maintainability)

**Rule of thumb**: If it describes "what the system does" → FR. If it describes "how well" → NFR.

### Requirement ID Prefixes

| Prefix | Source | Usage |
|--------|--------|-------|
| **FR-xxxx.x** | BRD | User-facing features (FR-1001+) |
| **NFR-xx** | BRD | Quality attributes (NFR-01 to NFR-07) |
| **DS-xxx** | Design System | Visual/UX requirements (DS-101 to DS-807) |
| **IMPL-xx** | UC Spec | Implementation details (not BRD-traceable) |

### Design System Traceability (DS-xxx)

Reference DS-xxx IDs from `specs/design/design-system.md`:

| Range | Category |
|-------|----------|
| DS-1xx | Page Structure & Layout |
| DS-2xx | Color Specifications |
| DS-3xx | Typography |
| DS-4xx | Component Specifications |
| DS-5xx | Interaction Patterns |
| DS-6xx | Data Visualization |
| DS-7xx | Accessibility |
| DS-8xx | Responsive Design |

**Example in plan.md**:
```markdown
### Layout Requirements
- [DS-101] Page background is light gray (#F4F6F8)
- [DS-109] Page container max-width is 1600px
- [DS-112] Left menu width is 384px fixed
```

### Technical Enabler Use Cases

Technical enablers like UC-01 (Frontend Framework) don't map directly to BRD FR numbers. Instead:

1. Add "BRD Traceability" section to spec explaining support relationship
2. Reference NFRs (NFR-05, NFR-07) and Design Principles
3. Use IMPL-xx prefix for implementation specifications
4. Reference DS-xxx for design system compliance

## Notes

- **Sequential for Dependencies**: Process in dependency order to ensure interfaces are defined
- **Parallelization**: Use cases UC-02 to UC-20 can run in parallel (all depend on UC-01)
- **Constitution Compliance**: Every plan must verify constitution principles
- **Research First**: Phase 0 research prevents technical debt
- **Tasks Separate**: This workflow stops at Phase 1; /tasks command generates tasks.md

# Generate Implementation Tasks

## How to Use This Prompt

**Purpose**: This prompt orchestrates the generation of actionable implementation tasks for enablers and use cases that have completed implementation plans.

**When to Use**:
- You have implementation plans (`plan.md`) in `specs/enablers/[ID]/` or `specs/use-cases/[ID]/`
- You want to create detailed task lists (`tasks.md`) for implementation
- Plans are complete (Phase 0 research + Phase 1 design done)

**How to Launch**:

```
Read the prompt file at .ai/example_prompts/generate-implementation-tasks.md and execute it.
Generate implementation tasks for all enablers and use cases that have plans.
```

**Alternative (specific items only)**:

```
Read the prompt file at .ai/example_prompts/generate-implementation-tasks.md and execute it.
Generate implementation tasks only for EN-06 and UC-01.
```

**Alternative (single item)**:

```
Read the prompt file at .ai/example_prompts/generate-implementation-tasks.md and execute it.
Generate implementation tasks for UC-05.
```

---

## Objective

Generate detailed, actionable implementation tasks following the tasks template structure at `.ai/2_templates/tasks-template.md`.

## Input Documents

Read these documents to understand scope:

1. **Tasks Template**: `.ai/2_templates/tasks-template.md` - Structure for task files
2. **Enablers Overview**: `specs/enablers/enablers-overview.md` - Lists all enablers
3. **Use Cases Overview**: `specs/use-cases/use-cases-overview.md` - Lists all use cases
4. **Constitution**: `../.ai_project_memory/constitution.md` - Core development principles
5. **Backend Constitution**: `../.ai_project_memory/constitution-backend.md` - Backend patterns
6. **Frontend Constitution**: `../.ai_project_memory/constitution-frontend.md` - Frontend patterns

## Prerequisites Check

Before generating tasks for an item:

1. **Plan exists**: `specs/[type]/[ID]/plan.md` must exist
2. **Research complete**: `specs/[type]/[ID]/research.md` should exist
3. **Design decisions documented**: Plan contains Phase 1 design decisions
4. **Dependencies have tasks**: Dependent items should have tasks.md first

## Output Requirements

For each enabler/use case with a plan, generate:

```
specs/[enablers|use-cases]/[ID]/
├── plan.md              # Must exist (input)
├── research.md          # Should exist (input)
└── tasks.md             # Generated by this workflow (output)
```

**Example**: `specs/enablers/EN-06/tasks.md`, `specs/use-cases/UC-01/tasks.md`

## Execution Strategy

**Constraint**: Process items in dependency order (same as plans).

**Resilience Design**: Stateless and idempotent - checks filesystem before each item.

### Phase 1: Initial Setup

1. **Read overview documents**:
   - Read `specs/enablers/enablers-overview.md`
   - Read `specs/use-cases/use-cases-overview.md`
   - Parse dependency graph to determine processing order

2. **Scan existing files**:
   - Check `specs/enablers/*/plan.md` for existing plans
   - Check `specs/enablers/*/tasks.md` for existing tasks
   - Check `specs/use-cases/*/plan.md` for existing plans
   - Check `specs/use-cases/*/tasks.md` for existing tasks
   - Identify items that have plans but no tasks

3. **Filter to actionable items**:
   - Only process items where `plan.md` exists
   - Skip items where `tasks.md` already exists (unless regeneration requested)

### Phase 2: Iterative Task Generation

For each item in dependency order:

1. **Verify prerequisites**:
   - Confirm `plan.md` exists
   - Skip if `tasks.md` already exists
   - Warn if `research.md` is missing

2. **Read plan and research**:
   - Read `specs/[type]/[ID]/plan.md`
   - Read `specs/[type]/[ID]/research.md` (if exists)
   - Extract: tech stack, project structure, design decisions, acceptance criteria

3. **Read dependent tasks** (if any):
   - Load tasks.md of dependencies to understand interfaces
   - Identify shared components or APIs to integrate with

4. **Generate tasks by category**:

   **Phase 3.1: Setup**
   - Project initialization
   - Dependency installation
   - Configuration files
   - Linting/formatting setup

   **Phase 3.2: Tests**
   - Contract tests for each API endpoint
   - Integration tests for user stories
   - Component tests for UI elements
   - Unit tests for business logic

   **Phase 3.3: Core Implementation**
   - Models and entities
   - Services and business logic
   - Controllers/endpoints
   - UI components
   - Input validation
   - Error handling

   **Phase 3.4: Integration**
   - Database connections
   - External service integrations
   - Middleware setup
   - Security configuration

   **Phase 3.5: Polish**
   - Additional unit tests
   - Performance optimization
   - Documentation updates
   - Code cleanup

5. **Apply task rules**:
   - Mark `[P]` for parallel execution (different files, no dependencies)
   - Ensure tests come before implementation
   - Number tasks sequentially (T001, T002...)
   - Include exact file paths in descriptions

6. **Write tasks.md**:
   - Create file at `specs/[type]/[ID]/tasks.md`
   - Include dependency graph
   - Include parallel execution examples

7. **Proceed to next item**

### Phase 3: Verification

1. Scan all item directories for `tasks.md` files
2. Compare against items with `plan.md`
3. Report any items with plans but no tasks
4. Validate task completeness per checklist

## Tasks Content Structure

Each `tasks.md` should follow this structure:

```markdown
# Tasks: [ID] - [Title]

**Input**: Design documents from `specs/[type]/[ID]/`
**Prerequisites**: plan.md (required), research.md
**Estimated Tasks**: [count]

## Phase 3.1: Setup
- [ ] T001 Create project structure per plan.md
- [ ] T002 Install dependencies: [list from plan]
- [ ] T003 [P] Configure ESLint and Prettier

## Phase 3.2: Tests
- [ ] T004 [P] Contract test [endpoint] in [path]
- [ ] T005 [P] Integration test [story] in [path]
...

## Phase 3.3: Core Implementation
- [ ] T008 [P] Create [model] in [path]
- [ ] T009 [P] Create [service] in [path]
...

## Phase 3.4: Integration
- [ ] T015 Connect [service] to [dependency]
...

## Phase 3.5: Polish
- [ ] T019 [P] Unit tests for [module] in [path]
- [ ] T020 Performance validation (<[target])
- [ ] T021 Update documentation
...

## Dependencies
- T004-T007 (tests) before T008-T014 (implementation)
- T008 blocks T009, T015
...

## Parallel Execution Example
```
# Launch T004-T007 together:
Task: "Contract test POST /api/v1/buildings in packages/API/test/buildings.e2e-spec.ts"
Task: "Contract test GET /api/v1/sensors in packages/API/test/sensors.e2e-spec.ts"
```

## Task Completeness Checklist
- [ ] All acceptance criteria from plan.md have corresponding tasks
- [ ] All acceptance criteria have corresponding test tasks
- [ ] Parallel tasks are truly independent (different files)
- [ ] Each task specifies exact file path
```

## Task Generation Rules

### From Plan.md

1. **Project Structure** → Setup tasks
   - Each directory to create
   - Each configuration file

2. **Design Decisions** → Implementation tasks
   - Each component/service
   - Each API endpoint
   - Each UI element

3. **Acceptance Criteria** → Test tasks
   - Each acceptance criterion → test task
   - Each performance target → validation task

### From Research.md

1. **Technology Decisions** → Setup tasks
   - Library installations
   - Configuration patterns

2. **Best Practices** → Integration tasks
   - Error handling patterns
   - Logging setup
   - Security measures

### Task Naming Convention

```
T001 [P] Create [component type] [name] in [exact/file/path.ext]
```

**Examples**:
- `T001 Create NestJS project structure per plan.md`
- `T002 [P] Install dependencies: @databricks/sql, @nestjs/config`
- `T003 [P] Configure ESLint in packages/API/.eslintrc.js`
- `T004 [P] Contract test GET /api/v1/buildings in packages/API/test/buildings.e2e-spec.ts`
- `T008 [P] Create BuildingsController in packages/API/src/controllers/buildings.controller.ts`

### Parallelization Rules

Mark `[P]` when:
- Tasks modify different files
- No data dependency between tasks
- Can be executed by separate agents simultaneously

Do NOT mark `[P]` when:
- Tasks modify the same file
- One task depends on output of another
- Tasks share state or configuration

## Processing Order

Based on project dependencies:

**Enablers First** (infrastructure before code):
```
EN-01 → EN-02 → EN-03 → EN-04 → EN-05 → EN-06 → EN-07 → EN-08
```

**Use Cases Second** (framework before features):
```
UC-01 → UC-02 to UC-20 (can be parallelized after UC-01)
```

## Quality Checklist

Before marking tasks complete:

- [ ] All acceptance criteria from plan.md have tasks
- [ ] Acceptance criteria have corresponding test tasks
- [ ] Parallel tasks are truly independent
- [ ] Each task has exact file path
- [ ] Dependencies are correctly identified
- [ ] Task count is reasonable (15-40 per item)
- [ ] No vague tasks ("implement feature")
- [ ] No same-file conflicts in parallel tasks

---

## Recovery from Interruption

If interrupted:
1. Restart the orchestrator
2. Filesystem scan identifies completed tasks.md files
3. Continue from next item without tasks
4. No manual intervention needed

---

## Notes

- **Exact Paths**: Every task must specify the exact file path
- **Parallelization**: Use [P] liberally for independent tasks
- **Dependencies First**: Process enablers before use cases
- **Constitution Compliance**: Tasks should follow coding standards
- **Commit Granularity**: Each task should be a committable unit of work
